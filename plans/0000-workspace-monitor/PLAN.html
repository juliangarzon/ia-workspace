Plan · 0000-workspace-monitor

AI Workspace Monitor (v1)

Build a read-only FastAPI dashboard that watches local AI coding workspaces via pluggable connectors. v1 ships: a connector framework, a Claude connector (quota windows, sessions, scheduled tasks), a read-only Docker connector, git/project monitoring, and a polling web dashboard with "IA", "Docker", and "Projects" tabs. Strictly observe-only; no writes to workspaces, no container control.

Stack: Python + FastAPI · JSON config, no DB · Deploy: Docker with state dirs + docker.sock mounted :ro · Spec: SPEC.html

Scope

In (v1)

  • Connector framework + registry
  • Claude connector: 5h / weekly / monthly quota windows
  • Sessions/agents with active-vs-stale status
  • Scheduled tasks list
  • Docker connector (read-only): status, uptime, CPU/mem, ports
  • Git/project monitoring (branch, dirty, last commit)
  • Web dashboard: IA / Docker / Projects tabs, client polling
  • In-process snapshot cache (TTL)
  • Docker image + compose with read-only mounts

Out

  • Any write or control (sessions, tasks, containers)
  • Process inspection / /proc reads
  • Push notifications / alerts / webhooks
  • Multi-user auth / RBAC
  • Historical warehouse / trend charts
  • Cost/billing in dollars, usage credits

Success Criteria

  • IA tab shows a Claude card with 5h/weekly/monthly windows (used, percent when limit configured, "resets in X"), refreshing without reload.
  • Window usage matches Claude's own widget within a few points when limits are configured.
  • Adding a connector = one new module under connectors/; aggregator, API, dashboard shell untouched.
  • Active sessions appear within one cache TTL; stale ones visibly distinguished.
  • Docker tab lists containers (name, image, status, uptime, CPU %, mem, ports); clean "no data" when socket unmounted.
  • Projects tab lists workspaces with git status; clean state when not a repo.
  • Service never writes to mounts, never issues a mutating Docker call, never returns raw transcript text.
  • Full snapshot rebuild over tens of projects completes well under the poll interval.
  • Missing/malformed sources degrade to partial data, no crash; parse-error counts surfaced on /healthz.

Assumptions

  • v1 is single-operator, single-host, bound to localhost. No auth is acceptable only under that constraint.
  • Config is JSON (no DB), holding poll interval, cache TTL, source paths, timezone, and optional per-window plan limits.
  • Plan limits are operator-supplied config; when absent, the UI shows raw usage (percent = null) rather than guessing caps.
  • "Git project monitoring" (in the v1 ask but only mentioned in passing in the spec) is treated as a real connector/task; its exact metric set is confirmed in Task 15 before building.
  • Codex connector is a best-effort fast-follow and explicitly out of v1 scope; the framework leaves room for it but v1 doesn't gate on it.
  • Full re-scan of transcripts behind a short cache TTL is acceptable for tens of projects; incremental offset-tracking is a perf follow-up only if scan latency becomes visible.
  • Riskiest tasks: #7b (streaming dedup by message.id — getting this wrong inflates every token count), #13/#14 (Docker socket access — the :ro guarantee and no-mutation invariant) and #8/#9 (quota-window math matching Claude's real reset semantics).
  • Token correctness depends on matching how existing tools parse ~/.claude: dedup partial records per message.id, drop zero-token turns, filter isSidechain, read costUSD directly, handle tiered cache fields. Confirmed in #7, enforced in #7b.
  • gitBranch is read from the JSONL per session; the git connector (#16) is for the Projects tab only, avoiding a per-session git fan-out.
