Spec · 0000-workspace-monitor

AI Workspace Monitor

A read-only web dashboard that watches local AI coding workspaces through pluggable connectors (Claude in v1), surfacing quota usage against the 5-hour / weekly / monthly limits, active sessions/agents, and scheduled tasks across tens of projects in one place. v1 also adds a Docker connector for read-only container monitoring (status, uptime, per-container CPU/memory, port mappings) in a dedicated "Docker" tab. Container control (start/stop/restart) is explicitly out of scope, consistent with the read-only v1 constraint.

Stack: Python + FastAPI · Architecture: connector-per-provider · Deploy: Docker with state files mounted read-only and /var/run/docker.sock mounted read-only · Scope: observation only — no writes to workspaces, no container control

Background

Julián runs many AI coding workspaces in parallel: Claude Code (desktop and CLI) and Codex, across tens of projects. Each tool writes its own state to the local filesystem (session files, transcripts, scheduled-task folders, a stats cache), but there is no single view that answers the questions that matter mid-day:

  • How much of my 5-hour / weekly / monthly quota have I consumed, and when does each window reset?
  • Which sessions and agents are running right now, and in which projects?
  • What scheduled tasks exist and when do they fire?

Today that means opening individual tools or grepping dotfiles by hand. The desired experience mirrors the plan-usage widget Claude already shows in its own UI — a percent-of-quota bar per window with a "resets in X" countdown — but aggregated across every workspace and presented in a browser.

The target widget Claude renders, which v1 reproduces, looks like:

  • 5-hour limit · 19% · resets 1h
  • Weekly · all models · 31% · resets 1d
  • Sonnet only · 11% · resets 1d
  • Usage credits · $0.00 of $90.00 (out of scope for v1 token tracking; see Non-Goals)

The monitor is organized around connectors: one self-contained module per AI provider that knows its own data source, its metrics, and how to present them. v1 ships the Claude connector; Codex and others follow the same interface later. The service runs in Docker with the relevant state directories mounted as read-only volumes, so it never mutates the workspaces it observes.

Goals

  • primary Show quota usage per window — 5-hour, weekly, monthly — as used vs. limit, percent consumed, and time-to-reset, mirroring Claude's own plan-usage widget. This is the headline metric, not a rolling projection.
  • primary Show active sessions and agents per project, derived from on-disk state files (Docker-friendly, no process inspection).
  • primary Ship a Claude connector as the first provider, behind a common connector interface so Codex / OpenAI / others slot in later without touching the core.
  • primary Ship a Docker connector (v1, read-only) that lists running containers with status, uptime, per-container CPU % and memory used/limit, and port mappings, surfaced in a dedicated "Docker" tab. Uses the same connector interface as the AI providers; reads the Docker socket mounted read-only.
  • List Claude scheduled tasks (one folder per routine) with their cadence.
  • Present everything in a single web dashboard (an "IA" tab with one card per AI connector, plus a "Docker" tab) that refreshes without manual reloads.
  • Stay strictly read-only: the monitor never writes to any observed workspace and never controls any container. Container start/stop/restart is explicitly out of scope — see Non-Goals and Security.
  • Scale comfortably to tens of projects (not hundreds) on a single host.
  • Aggregate Codex token usage on a best-effort basis once its connector lands: include it where the session files expose usage, degrade quietly where they don't.

Non-Goals

  • No push notifications, alerts, or webhooks in v1. Passive observation only; the user checks the dashboard.
  • No control actions: cannot start/stop sessions, kill agents, edit or trigger scheduled tasks, or start/stop/restart containers. The whole tool is read-only by design. Container start/stop/restart is explicitly out of scope for v1 — the Docker connector observes only. (Write access would require dropping the :ro flag on the socket mount, which opens full daemon control; see Security and Decisions Log.)
  • No process inspection (e.g. ps, reading /proc). Privileged Docker access for PID-namespace inspection is out of scope; AI state files are the single source of truth. Exception: the Docker connector reads /var/run/docker.sock — the standard, supported way to query container state, distinct from PID-namespace access. Mounted read-only in v1.
  • No multi-user auth, accounts, or RBAC. Single-operator, single-host tool.
  • No historical data warehouse or long-term retention beyond what the source files already keep.
  • No cost/billing reconciliation against provider invoices. Token counts come from local transcripts only.
  • Not designed for hundreds of projects or a fleet of remote hosts.

Design

Architecture

A FastAPI service loads one connector per AI provider, asks each to read its own mounted state and produce a normalized contribution, merges them into an in-memory snapshot, and serves both a server-rendered dashboard and a JSON API. The browser polls the API on an interval. No database: the filesystem is the store, and derived metrics are computed on demand with a short in-process cache.

A connector is the top-level abstraction. Each one is self-contained: it owns its source paths, its metrics, and its display contract. The core knows only the connector interface, so adding a provider means adding a module under connectors/ — nothing in the aggregator or API changes. v1 registers the Claude connector and the Docker connector; Codex and others register later. Internally a connector may use one or more collectors (one per source family) as an implementation detail; that is not the public abstraction.

The Docker connector is not an AI provider, but it implements the same Collector / connector interface: it reads its source (the Docker socket), normalizes the result into a snapshot contribution, and renders into its own dashboard tab. It surfaces in a dedicated "Docker" tab alongside the "IA" tab (one card per AI connector) and the git/projects tabs, rather than as a card inside "IA".

connectors/
  base.py     # Connector interface: id, sources, collect() -> ConnectorSnapshot
  claude.py   # v1: stats-cache + transcripts + sessions + scheduled-tasks
  docker.py   # v1: read-only container monitoring via Docker SDK over the socket
  codex.py    # later: best-effort token usage from ~/.codex/sessions
  # openai.py, ... future

Quota & Token Tracking Approach

The dashboard reproduces Claude's plan-usage widget: for each window it shows used vs. limit, percent consumed, and a "resets in X" countdown. It is a quota gauge, not a sliding projection. Two sources feed it:

  • stats-cache.json — cheap per-day activity rollups (message/session/tool counts). Drives the coarse window usage and is the fast path.
  • Transcript JSONL — each assistant turn carries a usage object, giving finer-grained token counts and per-turn timestamps for sub-day windows (the 5-hour bucket especially).

For each of the three fixed windows — 5-hour, weekly, monthly — the Claude connector computes:

  • used: token (or message) consumption attributed to the current window, bucketed by turn/day timestamp.
  • limit: the window cap, if known. Claude's real per-plan caps may not be exposed locally — see Open Decisions for the hardcode-vs-relative call.
  • percent: used / limit when a limit exists; null (show raw usage only) when it does not.
  • resets_at: the next window boundary, rendered as a "resets in X" countdown.

Token types captured from Claude transcripts: input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens. These are summed and also exposed broken out, since cache reads are priced differently and matter for the usage math.

Streaming dedup via message.id (required)

critical Claude Code writes multiple partial JSONL records per API call — one per streaming chunk. Only the last record for a given message.id carries the final, authoritative token counts; the earlier partials carry intermediate counts. Naively summing every record massively inflates token totals. The parser must therefore:

  • Track seen message.id values and keep only the last record per id (later records supersede earlier ones).
  • Drop turns where all four token counts (input, output, cache_creation, cache_read) sum to zero — these are non-usage or pre-stream records.

Cost: read costUSD directly

Claude Code writes costUSD directly onto each transcript record, so cost does not need to be computed from a pricing table. The connector reads costUSD as-is. A static pricing table is kept only as a fallback for older sessions whose records predate the costUSD field. (Dollar-cost surfacing in the UI remains deferred — see Non-Goals — but the field is parsed so it's available when that lands.)

Filter background work: isSidechain

Claude Code marks background and tool-spawned calls with isSidechain: true. These are not top-level user sessions and counting them as such inflates quota usage and the session list. The connector filters out (or tags separately) records and sessions where isSidechain is true when computing quota usage; they are never presented as standalone top-level sessions.

Cache tiering (newer API shape)

Newer Claude API responses split cache-creation tokens into tiers under message.usage.cache_creation:

  • message.usage.cache_creation.ephemeral_5m_input_tokens
  • message.usage.cache_creation.ephemeral_1h_input_tokens

These coexist with the flat cache_creation_input_tokens. The parser handles both shapes: prefer the flat field when present, else sum the tiered sub-fields.

Performance: prefilter before json.loads()

Before parsing a transcript line, cheaply test b'"usage":' in line and skip lines that fail it. The majority of JSONL lines carry no usage object, so this avoids the JSON-decode cost on the common case.

Codex (best-effort, later connector): same shape against ~/.codex/sessions/**/*.jsonl. Codex session files are JSONL with a leading session_meta record (cwd, model_provider, cli_version) followed by turn records that may carry token usage. Where usage is present it is aggregated; where the format omits it, the Codex connector contributes sessions/metadata only and reports its token data as unavailable rather than guessing.

Session & Agent Detection (state files only)

Confirmed approach: read on-disk state, never inspect processes. This is the only option that works cleanly inside Docker with read-only mounts and no privileged access.

  • Claude sessions live at ~/.claude/sessions/<pid>.json, one file per session, each carrying pid, sessionId, cwd, startedAt, kind (e.g. interactive), entrypoint (e.g. claude-desktop), and version.
  • Activity is derived from time since the last event (the latest turn timestamp / lastEventAt in the transcript JSONL for that session), not from process inspection. This is the approach proven by claude-code-dashboard. Three states, all thresholds configurable:
    • thinking — last event < 15s ago (thinking / waiting on the model).
    • active — last event < 60s ago.
    • idle / stale — last event > 60s ago; treated as inactive, since the file may be orphaned (see Edge Cases).
  • Records flagged isSidechain: true are excluded from the top-level session list (background/tool calls, not user sessions).
  • Project attribution comes from cwd on each session, matched against the transcript project folders.

Docker Monitoring (read-only, v1)

The Docker connector lists the host's containers and their live resource usage, read-only. It uses the Docker SDK for Python (the docker package) talking to the daemon over the Unix socket. The socket is mounted into the workspace-monitor container read-only:

-v /var/run/docker.sock:/var/run/docker.sock:ro

This is the standard pattern for monitoring Docker from inside a container. It is distinct from PID-namespace inspection (which the rest of the spec rules out): the socket is the daemon's own supported query interface, and the :ro flag keeps it observe-only. Per container the connector surfaces:

  • Identity & state: name, image, status (running / stopped / paused / exited), and uptime.
  • Resource usage: CPU percent and memory used vs. limit, from the daemon's stats stream (a single non-streaming sample per refresh).
  • Port mappings: host port → container port, with protocol.

Out of scope for v1: any container control (start / stop / restart). The connector never issues a mutating call, and the read-only socket mount makes that a structural guarantee, not just a code convention.

Git Monitoring (Projects tab vs. per-session branch)

Branch information comes from two distinct places, and conflating them is wasteful:

  • Per-session branch (AI tab): Claude Code writes gitBranch onto every transcript record. The branch a session is running on is read directly from the JSONL (Session.git_branch) — no git subprocess per session. This is cheap and always reflects the branch at the time of the turn.
  • Repo state (Projects tab): the git connector still runs git read-only against each configured project to show the current repo state — branch, dirty/clean, ahead/behind, last commit — across all configured repos, including those with no active AI session.

So the git connector exists for the Projects tab; the AI tab does not need it for branch-per-session. This keeps the per-poll cost down (no fan-out of git calls for every session).

Open Decisions

Q: How fresh must a session file be to count as "active"?
Options: (a) file mtime within N minutes; (b) startedAt within N hours; (c) cross-check against latest transcript turn timestamp for that cwd. Tradeoff: (a) is simplest but mtime may not update during long idle turns; (c) is most accurate but couples the session and transcript collectors. Leaning: start with (a) at a generous threshold, add (c) if false "stale" reports appear.
Q: Transcript parsing cost at tens of projects — full re-scan vs incremental?
Options: (a) re-read every .jsonl on each cache rebuild; (b) byte-offset / incremental tracking — store (path, mtime, line_count) per file and only read newly appended lines on re-scan. Tradeoff: (a) is trivial and fine for tens of projects if cache TTL is reasonable; (b) is necessary only if scan latency becomes visible over large transcript archives. Leaning: (a) behind a short TTL for v1; (b) as a Phase 2 perf optimization, not required for v1. Combined with the b'"usage":' prefilter, (a) should stay cheap at v1 scale.
Q: Are the window limits available locally, or do we hardcode / go relative?
The widget needs a limit per window to show a percent. Claude's real 5-hour / weekly caps may not be written to any local file. Options: (a) hardcode the known plan caps as config (e.g. a plan.yaml the operator sets to match their subscription); (b) ship without absolute limits and show raw usage only (percent = null), upgrading to (a) if caps surface. Tradeoff: (a) reproduces the real widget but goes stale when Anthropic changes plan limits and must be maintained by hand; (b) is always correct but less glanceable. Leaning: (a) behind operator-set config, falling back to (b)'s raw-usage display when no limit is configured. Blocks the percent display, not the rest of the spec.
Q: What defines each quota window's reset boundary (resets_at)?
Options: (a) fixed blocks anchored to first activity, mirroring Claude's real quota reset; (b) calendar-aligned (ISO week, calendar month) with a rolling 5-hour block. Tradeoff: (a) matches the "resets in X" the user sees in Claude's UI but requires knowing the anchor; (b) is unambiguous but the countdown won't line up with the real reset. Leaning: confirm Claude's actual reset semantics (esp. the 5-hour anchor) before locking; until then, calendar-align weekly/monthly and treat the 5-hour reset as the best-known approximation.
Q: Where does the scheduled-task cadence come from?
Each routine is a folder under ~/.claude/scheduled-tasks/. The schedule string lives in a per-folder config file whose exact name/format must be confirmed before next_run can be computed. Until then, list tasks by name and mark schedule as null.

Edge Cases & Error Handling

  • risk Orphaned session files: a crashed Claude process can leave a <pid>.json behind. Mitigation: freshness-based status; never claim "running" on file presence alone.
  • risk Partial / truncated JSONL: a transcript being written mid-turn may have a half line. Mitigation: parse line-by-line, skip lines that fail JSON decode, continue.
  • risk Schema drift: Claude/Codex change their on-disk formats between versions. Mitigation: tolerant parsing (missing keys default to zero/null), and a parse-error counter surfaced on /healthz rather than a hard crash.
  • Missing mounts: a volume (e.g. ~/.codex) not mounted. Mitigation: treat each source as optional; absent source contributes nothing and is flagged in the snapshot.
  • Docker socket absent or unreachable: the socket isn't mounted, or the daemon is down. Mitigation: the Docker connector reports available = false, the "Docker" tab renders a "no data" state, and docker_containers is empty — no crash, the rest of the dashboard is unaffected.
  • Container stats unavailable for a single container: a stopped/exited container has no live CPU/memory sample. Mitigation: list it with its status and uptime, leave the resource fields at zero/null rather than dropping the row.
  • Timezone / window edges: turns near a window boundary must bucket by a consistent timezone. Mitigation: parse timestamps as UTC, define windows in a single configured TZ.
  • Empty workspace: no sessions, no transcripts. Dashboard renders zeros, not errors.
  • Large transcripts: long-running projects produce big .jsonl files. Mitigation: stream-read; cap memory by summing as we go, never loading whole files into a list.

Security & Privacy

  • Sensitive content: transcripts contain full prompts and code, which may include secrets. The monitor reads them to count tokens but must never expose raw transcript text via the API or UI. Only aggregates and metadata leave the service.
  • Read-only by construction: all source directories are mounted read-only in Docker, so a bug cannot corrupt a workspace. This is a primary safety property, not a convenience.
  • risk Docker socket access: mounting /var/run/docker.sock with :ro is safe for monitoring — the connector can list containers and read stats, but cannot start, stop, or alter anything. The :ro flag is load-bearing. Removing it (mounting the socket read-write) hands the container full control of the Docker daemon, which is effectively root on the host. Treat any change that drops :ro as a security regression that must be reviewed, not a convenience tweak.
  • No auth in v1: acceptable only because the service binds to localhost / a private network on a single-operator host. Do not expose the port publicly. If that changes, auth moves from non-goal to blocker.
  • Path traversal: /api/projects/{name} takes a name from the client. Mitigation: resolve only against the discovered project set; reject anything not in the known list. Never build a filesystem path directly from user input.
  • Resource exhaustion: a malicious or huge file should not OOM the service. Mitigation: streaming reads and the snapshot cache bound work per request.

Testing Strategy

  • Unit: quota-window bucketing (turn/day timestamp → correct 5h/week/month window, including boundary timestamps); percent = used/limit with null-limit handling; resets_at computation; TokenBreakdown summation; stale-vs-active status logic.
  • Integration: the Claude connector against fixture directories that mirror real layouts (stats-cache.json, a multi-line transcript with usage objects, a <pid>.json session, a scheduled-task folder). Assert the per-connector contribution and the merged snapshot. A stub connector verifies the registry merges multiple providers without core changes.
  • Resilience: feed truncated JSONL, missing keys, and an absent mount; assert the service degrades to partial data and increments parse-error counters instead of crashing.
  • API: /api/snapshot shape and /api/projects/{name} rejection of unknown/traversal names; /api/docker shape; /healthz reports source availability.
  • Docker connector: against a fake/stub Docker client, assert containers map to DockerContainer rows (status, uptime, CPU/memory, port mappings) and that no mutating SDK call is ever issued. With the socket absent, assert available = false and an empty list rather than an exception.
  • Manual: run against the real local mounts read-only; eyeball that totals are in the same ballpark as Claude's own UI for a known session.

Success Criteria

  • The "IA" tab shows a Claude connector card with 5h / weekly / monthly windows, each rendering used, percent (when a limit is configured), and a "resets in X" countdown, refreshing without a manual reload.
  • Window usage for a known session matches Claude's own plan-usage widget within a small margin (percent within a few points when limits are configured).
  • Adding a second connector (e.g. Codex) requires only a new module under connectors/; the aggregator, API, and dashboard shell are untouched.
  • Active sessions appear within one cache TTL of their state file being written; stale ones are visibly distinguished.
  • Scheduled tasks are listed by name (cadence shown once the config format is confirmed).
  • The "Docker" tab lists running containers with name, image, status, uptime, CPU %, memory used/limit, and port mappings, refreshing on the same poll; with the socket unmounted it shows a clean "no data" state.
  • Service never writes to any mounted directory (verified by read-only mounts), never issues a mutating Docker call, and never returns raw transcript text.
  • A full snapshot rebuild over tens of projects completes fast enough that a poll feels instant (target: well under the poll interval).
  • Missing or malformed sources degrade gracefully: partial data, no crash.

Decisions Log

Q: Interface — web dashboard, CLI, or TUI?
Web dashboard. Best fit for an at-a-glance, always-open monitor and the only one that renders the cross-project rollups comfortably.
Q: How to detect sessions/agents — process inspection or state files?
State files only. Process inspection from inside Docker needs privileged access the user won't grant, and state files are the clean, mount-friendly source of truth.
Q: Language and framework?
Python + FastAPI. Fast to build a read-only JSON API plus a server-rendered page, strong JSON/JSONL parsing ergonomics, trivial to containerize.
Q: Top-level abstraction — collectors or connectors?
Connectors. One self-contained module per AI provider (connectors/claude.py, later connectors/codex.py), each implementing a shared interface and owning its source, metrics, and display. The dashboard surfaces this directly as an "IA" tab with one card per connector. Rejected the flat "collectors" framing as the public abstraction because it leaked provider concerns into the core; collectors survive only as an internal detail inside a connector. This is what makes adding OpenAI/others a drop-in.
Q: Headline metric — quota windows or rolling token sums?
Quota windows. The dashboard reproduces Claude's plan-usage widget (used vs. limit, percent, "resets in X") rather than a sliding projection, because that is the at-a-glance view the user actually reasons about. Rolling raw sums are kept only as the underlying used breakdown.
Q: Token source — stats-cache or transcripts?
Both, by role. stats-cache.json gives cheap per-day rollups for the coarse window usage (the fast path); transcripts (each turn's usage object) provide finer-grained, per-turn counts for sub-day windows like the 5-hour bucket. This mirrors how Claude computes its own stats.
Q: Codex token tracking — v1 guarantee or best-effort?
Best-effort, and behind its own (later) connector. Codex session-file formats vary and don't always expose token usage, so promising exact Codex totals in v1 would be dishonest. Where usage is present the connector aggregates it; where it isn't, the card shows sessions/metadata and marks token data unavailable. v1 ships the Claude connector; Codex is a fast-follow that doesn't gate v1.
Q: Notifications in v1?
No. v1 is passive observation; the user checks the dashboard. Push/alerts deferred to avoid scope creep before the core metrics are trusted.
Q: Read-only or control actions?
Read-only. Safer, simpler, and enforced by read-only Docker mounts. Control surface deferred.
Q: Docker monitoring — read-only socket mount or write access?
Read-only socket mount (-v /var/run/docker.sock:/var/run/docker.sock:ro), via the Docker SDK for Python. Ruled out write (read-write) access: a writable socket is full control of the Docker daemon, which is effectively host root, and it would break the read-only-by-construction guarantee that the rest of the tool relies on. Container start/stop/restart is therefore out of scope for v1. The connector reads container list + stats + ports only; the :ro flag makes "observe-only" a structural property, not a code convention. Chose the socket over PID-namespace inspection because the socket is the daemon's own supported query interface and mounts cleanly without privileged access.
Q: Storage — database or filesystem?
No database. The mounted state files are the store; derived metrics live in an in-process snapshot cache. Tens of projects don't justify the operational weight of a DB.

Prior Art & References

The Claude token-tracking and session-activity approach draws on existing open-source tools that parse the same ~/.claude transcript format. Key findings adopted from them:

  • claude-code-dashboard — time-based session activity (derive thinking/active/idle from lastEventAt rather than process inspection).
  • Community token-counting tools — the streaming-dedup insight: multiple partial JSONL records per message.id, only the last carries final counts; drop zero-token turns. Also the costUSD-is-already-written and isSidechain-filtering observations, the tiered cache_creation shape, and the b'"usage":' prefilter for cheap line skipping.

These are credited because v1's correctness (especially not inflating token counts) depends on matching how these tools read the format. Where this spec diverges, it's noted inline.
